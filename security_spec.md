# Security Spec

## 1. Data Invariants
- A `users` document cannot be created without a valid UID that matches `request.auth.uid`.
- A user's `balance` can only be updated by an admin, OR by exactly 10 during the once-a-day check-in.
- `role` must be 'user' or 'admin'. Self-registration maps strictly to 'user'.
- `transactions` amount must be positive. Once completed, they are immutable.
- `orders` must belong to the user.
- `redeem_codes` can only be generated by admins. A user redeeming a code must flip `used` from false to true.

## 2. The "Dirty Dozen" Payloads
1. **Shadow Update (Ghost Field):** Injecting `isAdmin: true` into a profile update.
2. **Email Spoofing:** Sending profile fields with an unverified admin email.
3. **PII Blanket:** Attempting to read `/users/{otherUser}` explicitly.
4. **Denial of Wallet:** Creating a user with a 1.5MB string in the `bankDetails.accountNumber` field.
5. **State Shortcutting:** User updates an order `status` to 'completed'.
6. **Balance Manipulation:** User updates `balance` directly by +1000.
7. **Identity Spoofing:** Creating a transaction where `userId` does not match `request.auth.uid`.
8. **Value Poisoning:** Changing `balance` type to string or boolean.
9. **Role Escalation:** Self-registering with `role: admin`.
10. **Order Re-creation:** Updating `createdAt` during a normal update.
11. **Redemption Replay:** Updating a `redeem_code` that is already `used: true`.
12. **Unverified Auth Gate:** Accessing resources with `email_verified: false` where verification is mandated.

## 3. The Test Runner
A `firestore.rules.test.ts` file will be created to verify these constraints.
